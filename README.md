# swift-embedded-ps2

> **Embedded Swift → WebAssembly → C89 (w2c2) → MIPS R5900 (PS2 Emotion Engine)**

Run Swift on the PlayStation 2 — no native Swift MIPS backend required.

```
Sources/PS2Demo/main.swift
        │
        │  swift build --swift-sdk swift-6.2-RELEASE_wasm-embedded
        ▼
build/PS2Demo.wasm          (wasm32-unknown-none-wasm, ~30 KB)
        │
        │  w2c2 -p PS2Demo.wasm PS2Demo.c
        ▼
build/PS2Demo.c / .h        (portable C89, ~500 KB before optimisation)
        │
        │  mips64r5900el-ps2-elf-gcc ... -o PS2Demo.elf
        ▼
build/PS2Demo.elf           (EE ELF for PCSX2 / ps2link)
```

## Why this works

The PS2's Emotion Engine is a **MIPS R5900** — a little-endian custom MIPS III core. WASM is also little-endian and 32-bit. [w2c2](https://github.com/turbolent/w2c2) transpiles WASM to C89, which `mips64r5900el-ps2-elf-gcc` (part of ps2dev, now at GCC 15.2) can compile for the EE with no issues.

This is the same approach [turbolent used to run Swift on PowerPC Mac OS 9 and NeXTSTEP](https://forums.swift.org/t/retrocomputing-what-would-it-take-to-add-powerpc-darwin9-support/71797/8) — zero changes to the Swift compiler or LLVM required.

## Prerequisites

### Step 1 — Embedded Swift → WASM (on your Mac/Linux host)

Install Swift 6.2+ and the Embedded WASM SDK:

```sh
# Install swiftly (Swift version manager)
curl -O https://download.swift.org/swiftly/darwin/swiftly.pkg
installer -pkg swiftly.pkg -target CurrentUserHomeDirectory
~/.swiftly/bin/swiftly init
swiftly install 6.2.0
swiftly use 6.2.0

# Install Embedded Swift WASM SDK
swift sdk install \
  https://download.swift.org/swift-6.2-release/wasm-sdk/swift-6.2-RELEASE/swift-6.2-RELEASE_wasm-embedded.artifactbundle.tar.gz
```

### Steps 2 + 3 — w2c2 + ps2dev GCC (via Docker, easiest)

```sh
# Pull the official ps2dev image (includes mips64r5900el-ps2-elf-gcc)
docker pull ps2dev/ps2dev:latest

# w2c2 is built inside the container by make docker-elf (see below)
```

Or install w2c2 locally:

```sh
git clone --depth 1 https://github.com/turbolent/w2c2
cmake -S w2c2 -B w2c2-build -DCMAKE_BUILD_TYPE=Release
cmake --build w2c2-build -j$(nproc)
sudo cmake --install w2c2-build
```

## Building

### Quickest path (host Swift + Docker for ps2dev)

```sh
# Step 1: Compile Swift to WASM on your host
make wasm

# Steps 2+3: w2c2 transpile + ps2-gcc link, all inside ps2dev Docker
make docker-elf
```

### All steps locally (if you have all tools installed)

```sh
make all
```

### Individual steps

```sh
make wasm      # → build/PS2Demo.wasm
make c         # → build/PS2Demo.c, build/PS2Demo.h
make elf       # → build/PS2Demo.elf
```

## Key concepts

### Embedded Swift constraints

The Swift source uses only:
- `@_silgen_name` / `@_cdecl` for import/export
- Fixed-width integer types (`Int32`, `UInt8`, etc.)
- No `String`, no `Array`, no `print()` — all I/O goes through imported C functions
- `Float` technically works on EE but the FPU is non-IEEE 754 (32-bit only, no doubles)

### The import/export contract

Swift declares imports with `@_silgen_name`:

```swift
@_silgen_name("ps2_print")
func ps2_print(_ msg: UnsafePointer<UInt8>, _ len: Int32)
```

w2c2 lowers this to a C function pointer in the generated module:

```c
// In PS2Demo.h (generated by w2c2):
extern void (*f_PS2Demo_ps2_print)(U32 ptr_offset, I32 len);
```

`glue.c` assigns the real implementation before calling `init()`:

```c
f_PS2Demo_ps2_print = host_ps2_print;
```

### WASM linear memory as host array

w2c2 exposes the WASM linear memory as a `wasmMemory*`. We allocate it statically in `glue.c` — 2 MB fits comfortably in the PS2's 32 MB EE RAM and is more than enough for Embedded Swift output:

```c
static U8 wasm_heap[32 * 65536];   // 2 MB
```

All pointer arguments from Swift arrive as `U32` offsets into this array, not host pointers.

### w2c2 output file naming

After running `w2c2 -p PS2Demo.wasm PS2Demo.c`, the generated code uses a module prefix derived from the `.wasm` filename. The function pointer extern names will be `f_PS2Demo_*`. If your module name differs, adjust the `extern` declarations in `glue.c` accordingly.

## Running in PCSX2

1. Build `build/PS2Demo.elf`
2. Open PCSX2
3. **File → Run ELF** → select `PS2Demo.elf`
4. Open **Debug → EE Console** to see the stdout output from `glue.c`

For the plasma demo to render visually, hook up `host_ps2_draw_pixel` in `glue.c` to a real gsKit draw call (see comments in the function).

## Adding gsKit graphics

To render actual pixels, install gsKit in the ps2dev environment and update `glue.c`:

```c
#include <gsKit.h>
#include <dmaKit.h>

static GSGLOBAL* gsGlobal;

// In main(), before calling PS2Demo_init():
gsGlobal = gsKit_init_global();
dmaKit_init(D_CTRL_RELE_OFF, D_CTRL_MFD_OFF, D_CTRL_STS_UNSPEC,
            D_CTRL_STD_OFF, D_CTRL_RCYC_8, 1 << DMA_CHANNEL_GIF);
dmaKit_chan_init(DMA_CHANNEL_GIF);
gsKit_init_screen(gsGlobal);
gsKit_mode_switch(gsGlobal, GS_MODE_NTSC);

// In host_ps2_draw_pixel():
gsKit_prim_point(gsGlobal, (float)x, (float)y, 1,
                 GS_SETREG_RGBAQ(cur_r, cur_g, cur_b, 0x80, 0.0f));

// In host_ps2_vsync():
gsKit_queue_exec(gsGlobal);
gsKit_sync_flip(gsGlobal);
```

## R5900 quirks to be aware of

| Quirk | Impact on this pipeline |
|---|---|
| No `double` FPU (32-bit only, non-IEEE 754) | Avoid `Double` in Swift; w2c2 emits `F64` via `double` C type which the EE FPU emulates in software — slow | 
| Short loop bug | ps2dev GCC already patches this; w2c2 C output uses standard loops that GCC handles correctly |
| No `ll`/`sc` (load-linked / store-conditional) | Embedded Swift is single-threaded; no atomics needed |
| 128-bit GPRs / MMI SIMD | Not used by w2c2 C output; GCC may use them for optimisation automatically |
| Little-endian | Matches WASM byte order; `w2c2_base.h` memory macros use `memcpy` which is endian-correct |
